## Project Overview

MUNify STOCK (System for Tracking and Organization of Containers and Kartons) is inventory management software for Model United Nations conferences, part of the MUNify ecosystem by Deutsche Model United Nations (DMUN) e.V.

## Commands

```bash
bun run dev              # Start dev server + Docker (PostgreSQL)
bun run dev:server       # Dev server only (no Docker)
bun run build            # Production build
bun run check            # Svelte + TypeScript type checking
bun run lint             # Prettier + ESLint check
bun run format           # Auto-format all files
bun run generate:client  # Generate typed GraphQL client (requires running dev server)
bun run db:push          # Push Drizzle schema to database
bun run db:migrate       # Run Drizzle migrations
bun run db:studio        # Open Drizzle Studio (DB GUI)
```

## Architecture

### Stack

- **Frontend**: SvelteKit 2 + Svelte 5 (runes mode), Tailwind CSS 4, DaisyUI 5, DMUN Corporate Identity theme
- **Backend**: GraphQL via rumble v0.16 (Pothos + GraphQL Yoga), served at `/api/graphql`
- **Database**: PostgreSQL via Drizzle ORM v1 (snake_case mapping, nanoid IDs)
- **Auth**: OIDC via `@logto/sveltekit` (Logto) — supports browser login + M2M bearer tokens (JWKS validation via `jose`)
- **GraphQL Client**: rumble's built-in client generator (URQL-based, Svelte runes support)
- **i18n**: Paraglide (baseLocale: `de`, locales: `de`, `en`, cookie-based strategy)

### Key Directories

- `src/api/db/` — Drizzle schema (`schema.ts`), relations (`relations.ts`), client (`db.ts`)
- `src/api/handlers/` — GraphQL handlers (one file per entity), registered via `register.ts`
- `src/api/services/` — Admin email check, audit log
- `src/api/context.ts` — GraphQL context (Logto user extraction, role checking, M2M token support)
- `src/api/rumble.ts` — Rumble initialization (exports schemaBuilder, abilityBuilder, etc.)
- `src/lib/config/` — Zod-validated environment config (public.ts, private.ts)
- `src/lib/components/` — Reusable Svelte components (ThemeSwitcher, LanguageSwitcher, SearchModal, FormFieldset, QrCodeDisplay, NavMenu/\*, SideNavigationDrawer, Footer)
- `src/lib/generated-client/` — Auto-generated rumble GraphQL client (gitignored, regenerate with `bun run generate:client`)
- `src/lib/paraglide/` — Auto-generated i18n runtime (gitignored, regenerated by Vite plugin)
- `src/lib/utils/` — Utilities (theme switcher)
- `src/lib/helpers/` — nanoid, graphql helper
- `src/routes/app/` — Protected app routes (requires Logto login)
- `src/routes/(public)/` — Public landing page
- `src/routes/api/graphql/` — GraphQL endpoint
- `messages/` — i18n message files (`de.json`, `en.json`)

### Handler Pattern

Each handler in `src/api/handlers/` follows this pattern:

1. Call `basics(tableName)` for standard CRUD (object type + query + pubsub)
2. Set ability rules via `abilityBuilder.tableName.allow('read').when(...)`
3. Add custom mutations via `schemaBuilder.mutationFields(...)`
4. Export the ref for cross-handler use
5. Import the handler in `register.ts`

Delete mutations use `t.field({ type: 'Boolean' })` — these are not compatible with the rumble client's field selection, use direct `fetch` to `/api/graphql` instead.

### GraphQL Client Pattern

Pages use the rumble-generated client (`$lib/generated-client/client`) for all data:

```ts
import { client } from '$lib/generated-client/client';

// Live query (reactive, auto-subscribes)
const items = client.liveQuery.items({ id: true, name: true, ... });
items.subscribe((v) => { if (v) myState = v; });

// Single entity by ID
client.liveQuery.item({ __args: { id: someId }, id: true, name: true });

// Mutation
await client.mutate.createItem({ __args: { name, ... }, id: true });
```

After changing the schema or handlers, regenerate the client:

```bash
bun run generate:client  # dev server must be running
```

### Auth Flow

- Logto SDK (`handleLogto`) in `hooks.server.ts` handles session auth and `/callback` route
- Auth guard in hooks protects `/app` routes (redirects to Logto sign-in)
- User upserted to DB on each authenticated `/app` request
- M2M auth via `Authorization: Bearer <token>` header, validated with JWKS (`jose`)
- Roles: "admin" (full access), "member" (standard access)
- Access gated by `ACCESS_DOMAIN_WHITELIST` (e.g., `@dmun.de`)
- Context provides `mustBeLoggedIn()`, `hasRole()` for GraphQL resolvers (supports both session and M2M)
- `isGlobalAdmin()` checks admin role OR whitelisted email/domain

### i18n

- Messages in `messages/de.json` and `messages/en.json`
- Import as `import * as m from '$lib/paraglide/messages'`
- Use `m.keyName()` in templates, `m.keyName({ param: value })` for interpolation
- Cookie-based locale switching via `LanguageSwitcher` component
- Paraglide middleware in hooks handles server-side locale

## Commit Convention

Follow conventional commits with emoji prefixes and scope in parentheses:

```
✨ feat (Frontend): implement login button and modal
🐞 fix (General): address error in login body
📚 docs (Backend): add examples
```

Prefixes: `✨ feat`, `⬆️ improvement`, `🐞 fix`, `🎨 style`, `🗺️ i18n`, `🚧 wip`, `🧼 format & lint`, `🔤 typing`, `📚 docs`, `🚀 perf`, `🧪 test`, `🧹 chore`, `♻️ refactor`, `🏗️ build`, `🚦 ci`, `↩️ revert`, `🔀 merge`
